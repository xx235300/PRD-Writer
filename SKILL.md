## 工作模式概览

本技能通过**三阶段工作流**生成结构化 PRD：

| 阶段 | 名称 | 目标 |
|------|------|------|
| 阶段一 | Context Gathering | 收集并确认需求信息，识别输入模式 |
| 阶段二 | Refinement & Structure | 按 PRD 章节结构组织内容，应用 MoSCoW 优先级 |
| 阶段三 | Reader Testing | 验证可读性，收集反馈，迭代更新 |

---

## 阶段一：Context Gathering（需求收集）

### 识别输入模式

根据用户输入形式判断当前模式：

| 模式 | 判断依据 | 初始动作 |
|------|---------|---------|
| **模式A** | 用户用自然语言描述需求想法，无现成文档 | 直接进入阶段一，逐一提问收集 |
| **模式B** | 用户粘贴了一段文字（PRD草稿/需求描述/功能列表） | 先分析内容缺口，再针对性提问 |
| **模式C** | 用户提供了飞书文档链接 | 用 `feishu_fetch_doc` 读取，再针对性提问 |

**模式识别提示语**：
- 模式A典型句式：「我想做…」「我们需要开发…」「帮我规划…」
- 模式B典型句式：「帮我完善下这个PRD」「这是我写的需求，请优化」「草稿如下：…」
- 模式C典型句式：「基于这个文档生成PRD」「帮我把这个链接转成PRD」

### 收集维度

无论哪种模式，阶段一必须确认以下关键信息：

| 维度 | 对应章节 | 典型问题 |
|------|---------|---------|
| 项目背景与目标 | 项目概述 | 这个产品解决什么问题？目标是什么？ |
| 目标用户 | 用户分析 | 谁会用这个功能？C端/B端？主要和次要用户？ |
| 核心功能范围 | 需求详述 | 最核心的3个功能是什么？哪些是必须有的？ |
| 优先级判断 | 需求详述 | 哪些是必须有（MUST）？哪些可以放下一版？ |
| 成功指标 | 成功指标 | 希望通过这个功能达成什么可量化的结果？ |
| 验收标准 | 验收标准 | 怎样算「做好了」？关键场景是什么？ |
| 技术约束 | 技术约束 | 有没有技术限制？依赖哪些系统或第三方服务？ |
| 输出方式 | — | 输出到飞书文档还是直接发给我？ |

### 提问策略

- **模式A**：按维度逐一提问，每次 2-3 个问题，避免一次问太多造成压力
- **模式B**：只问缺失章节和关键缺口，已完整的内容跳过
- **模式C**：只问读取文档后仍无法填补的信息缺口

### 需求确认清单

阶段一结束时，向用户输出一份确认清单：

```markdown
## ✅ 需求收集清单

请确认以下信息，我将基于此生成 PRD：

- **项目名称**：[名称]
- **项目背景**：[背景描述]
- **目标用户**：主要用户… / 次要用户…
- **核心功能（P0/Must）**：[功能A]、[功能B]、[功能C]
- **次要功能（P1/Should）**：[功能D]、[功能E]
- **可选功能（P2/Could）**：[功能F]
- **成功指标**：[指标1] ≤ [目标值]，[指标2] ≥ [目标值]
- **技术约束**：[无特殊限制 / 列明约束]
- **文档语言**：[中文 / 英文 / 双语]
- **输出方式**：[对话内输出 / 飞书云文档]

请确认或补充，以上信息将作为 PRD 生成依据。
```

## 阶段二：Refinement & Structure（完善与结构化）

### PRD 章节生成顺序与优先级

| 生成顺序 | 章节 | 优先级 | 说明 |
|---------|------|--------|------|
| 1 | 项目概述 | P0（Must） | 背景、目标、范围边界、价值主张 |
| 2 | 用户分析 | P0（Must） | 用户画像、用户旅程图 |
| 3 | 需求详述 | P0（Must） | 功能清单（MoSCoW）、功能详述、业务规则 |
| 4 | 成功指标 | P1（Should） | 北极星指标、领先指标（Leading）、滞后指标（Lagging） |
| 5 | 验收标准 | P1（Should） | Given/When/Then 格式用例 |
| 6 | 技术约束 | P2（Could） | 技术栈、集成依赖、安全合规 |
| 7 | 附录 | P2（Could） | 术语表、版本记录、联系人 |

> **说明**：P0 章节为必填，用户未提供的信息用 `[待确认]` 占位并标注状态。P1/P2 章节若信息不足可简化，但需注明 `[可选补充]`。

### MoSCoW 优先级体系

在需求详述章节的功能清单中，每个功能必须标注优先级：

| 优先级 | 含义 | 说明 |
|--------|------|------|
| **P0 / Must** | 必须有 | 系统无法正常使用，MVP 底线 |
| **P1 / Should** | 应该有 | 重要但不影响主流程完成 |
| **P2 / Could** | 可以有 | 提升体验，本次可选 |
| **Won't** | 这次不做 | 列入未来规划，本次明确排除 |

**功能清单示例**：

```markdown
| 功能模块 | 功能名称 | 功能描述 | 优先级 | 备注 |
|----------|----------|----------|--------|------|
| 反馈提交 | 反馈表单 | 用户可提交文字反馈并选择问题类型 | P0 | Must，必须有 |
| 反馈提交 | 历史反馈 | 用户可查看自己提交的历史反馈列表 | P0 | Must，必须有 |
| 后台管理 | 反馈列表 | 客服可查看、筛选所有反馈记录 | P1 | Should，应有 |
| 数据分析 | 统计面板 | 展示反馈量和问题类型分布 | P2 | Could，可以有 |
| 用户通知 | 邮件通知 | 反馈状态变更时发送邮件通知 | Won't | 本次不做 |
```

### Given/When/Then 验收标准格式

每个 P0 和 P1 功能必须编写 2-5 个验收用例，使用 Given/When/Then 格式：

```markdown
### 3.2.1 反馈表单提交

**验收标准 1：正常提交反馈**

- **Given**：用户已登录，且在产品使用中
- **When**：用户选择问题类型为「功能异常」，填写描述「提交按钮无法点击」，上传 1 张截图，点击「提交」按钮
- **Then**：系统显示「提交成功」提示，生成反馈编号，弹窗在 3 秒后自动关闭

**验收标准 2：必填项校验**

- **Given**：用户打开反馈表单
- **When**：用户未选择问题类型，直接点击「提交」按钮
- **Then**：系统显示「请选择问题类型」的提示，表单不提交，当前填写内容保留

**验收标准 3：附件格式校验**

- **Given**：用户点击「上传截图」按钮
- **When**：用户选择了一个 10MB 的 PDF 文件
- **Then**：系统显示「仅支持 JPG、PNG、GIF 格式，单个文件不超过 5MB」的提示，不上传该文件
```

**Given/When/Then 写作规则**：
- **Given**：描述用户进入操作前的状态，不描述系统内部状态
- **When**：描述具体的用户操作序列，包含关键参数
- **Then**：描述可观测、可验证的结果（界面变化、数据变化、通知等）
- 每个验收用例只验证**一个**结果，避免堆叠多个 Then
- 每个 P0 功能建议 3-5 个用例（覆盖主流程 + 主要异常）

### 成功指标格式

```markdown
## 4.1 北极星指标

| 指标名称 | 定义 | 目标值 | 类型 |
|----------|------|--------|------|
| 反馈处理时长 | 从用户提交到首次处理的时间 | ≤ 24h | Lagging |
| 反馈提交率 | 提交用户 / DAU | ≥ 5% | Leading |

## 4.2 支撑指标

| 指标名称 | 定义 | 目标值 | 类型 |
|----------|------|--------|------|
| 归类准确率 | 自动归类正确的反馈数 / 总反馈数 | ≥ 85% | Lagging |
| 客服响应速度 | 客服首次查看反馈的平均时长 | ≤ 2h | Leading |
```

## 阶段三：Reader Testing（可读性验证）

### 文档结构概览

PRD 生成完成后，首先向用户提供文档结构概览：

```markdown
## PRD 生成完成

**项目名称**：《[项目名称] 产品需求文档 v0.9》
**文档语言**：[中文 / 英文]
**生成时间**：[时间戳]

### 文档结构

- 第一章 项目概述 ✅ 100%
- 第二章 用户分析 ✅ 100%
- 第三章 需求详述 ✅ 100%（含 P0/P1/P2 优先级标注）
- 第四章 成功指标 ⚠️ 部分指标待确认
- 第五章 验收标准 ✅ 100%（含 N 个 Given/When/Then 用例）
- 第六章 技术约束 ✅ 100%
- 第七章 附录 — 可选补充

### ⚠️ 待确认事项

1. [具体待确认项 1]
2. [具体待确认项 2]

---

请确认是否有需要调整的方向，我可以针对性更新指定章节。
```

### 完整性检查清单

请用户逐项确认：

```markdown
## PRD 完整性检查

请确认以下内容：

1. **项目概述**：背景描述是否符合预期？目标是否可量化？
2. **用户分析**：目标用户是否覆盖全面？用户旅程是否完整？
3. **需求详述**：
   - P0/P1/P2 分级是否符合预期？
   - 功能详述中的业务规则是否准确？
   - 是否有遗漏的核心场景？
4. **成功指标**：
   - 指标目标值是否合理？
   - 领先指标和滞后指标的逻辑关系是否成立？
5. **验收标准**：
   - Given/When/Then 格式是否清晰？
   - 主流程和主要异常流程是否都有覆盖？
6. **技术约束**：技术栈和集成依赖是否准确？有无遗漏？
7. **附录**：术语表、联系人是否需要补充？
```

### 关键判断点确认

主动询问用户关键决策点：

- "P0 功能清单是否符合预期？如果开发时间紧张，你会优先保哪3个？"
- "成功指标中『反馈处理时长』目标值设为24h，你希望更激进（4h）还是更保守（48h）？"
- "是否需要支持移动端？这会影响技术架构选择。"

### 迭代更新规则

根据用户反馈更新指定章节，更新时标注变更记录：

```markdown
## 更新记录

- `[v0.9 → v0.91]` 第四章成功指标：首次响应时长目标值从 24h 调整为 4h
- `[v0.9 → v0.91]` 第三章需求详述：新增「邮件通知」功能，优先级 P1
- `[v0.91 → v0.92]` 第五章验收标准：补充「附件超限」异常场景验收用例
```

## 输入模式处理详解

### 模式A：用户一句话需求 → 从零生成完整 PRD

**典型对话**：

```
用户：我想做一个用户反馈系统
AI：[识别为模式A，进入阶段一]

我需要了解一些信息来帮你生成完整的 PRD：

1. **目标用户**：这个系统给谁用？（C端用户提交反馈？客服处理反馈？两者都要？）
2. **核心目标**：希望通过这个系统达成什么？（提高处理效率？收集更多反馈？）
3. **范围边界**：有没有时间或技术限制？（几周上线？用现有技术栈？）

请告诉我，我会基于这些信息生成完整的 PRD。
```

### 模式B：用户粘贴待优化 PRD → 分析缺口并完善

**典型对话**：

```
用户：[粘贴了一段300字的需求描述]
AI：[分析内容后，识别缺口]

我看到了您的需求描述，核心功能已清晰。
以下章节需要补充：

- ❌ 验收标准：缺少具体验收条件（需要 Given/When/Then 用例）
- ⚠️ 成功指标：只提到「提高效率」，没有量化指标
- ⚠️ 技术约束：没有提及技术限制或外部依赖

请提供：
1. 核心功能的关键验收场景（3-5 个）
2. 希望达成什么可量化的结果？
3. 有没有技术限制或第三方依赖？

我会基于这些信息生成完整 PRD。
```

### 模式C：用户提供飞书文档链接 → 读取后生成 PRD

**典型对话**：

```
用户：https://xxx.feishu.cn/doc/xxx 基于这个帮我生成 PRD
AI：[使用 feishu_fetch_doc 读取文档]

已读取您的需求文档，涵盖了主要功能模块。
以下信息仍需要补充：

1. 目标用户是否包括客服侧？（文档中未提及客服操作需求）
2. 成功指标中「处理时长」目标值是多少？
3. 是否需要移动端支持？

请告诉我以上信息，我将生成完整 PRD。
```

## 输出方式

### 方式一：对话内输出（默认）

直接在对话中输出 Markdown 格式的完整 PRD。

每章结尾标注完成度：

```markdown
## 第一章、项目概述 ✅ 100%

...

## 第四章、成功指标 ⚠️ P1（部分指标数值待确认）

| 指标名称 | 定义 | 目标值 | 状态 |
|----------|------|--------|------|
| 首次响应时长 | 提交到首次回复的时间 | ≤ 24h | ✅ 确认 |
| 反馈提交率 | 提交用户/D AU | ≥ 5% | ⚠️ 待确认目标值 |
```

**完成度标注规则**：
- ✅ **100%**：内容完整，用户已确认
- ⚠️ **P1/P2**：部分内容依赖用户确认或补充
- ❌ **待补充**：章节缺失或内容严重不完整

### 方式二：飞书云文档输出

用户明确指定飞书文档时，使用 `feishu_create_doc` 创建文档：

```markdown
PRD 生成完成！

- 文档标题：《[项目名称] 产品需求文档 v0.9》
- 飞书链接：https://xxx.feishu.cn/doc/xxx

请确认内容，我会根据您的反馈更新到 v1.0。
```

## 重要原则

1. **不猜测未确认的信息**：用户未确认的内容用 `[待确认]` 占位，不自行填充
2. **MoSCoW 优先级必须与用户确认**：P0/P1/P2 的划分需要用户参与判断，不替用户做主
3. **Given/When/Then 不过度展开**：每个 P0 功能 3-5 个核心用例，覆盖主流程和主要异常
4. **语言一致性**：同一份 PRD 中术语、格式、语气保持一致
5. **章节可独立确认**：用户可以只确认某几个章节，无需一次性确认整份文档
6. **不使用 sessions_spawn**：所有工作在当前会话内完成
