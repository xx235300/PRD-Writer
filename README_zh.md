## ✨ 亮点

**PRD-Writer** 是一款基于 AI 的产品需求文档写作助手，将产品经理从繁琐的文档写作中解放出来，把时间留给真正的产品思考。

只需输入需求想法或待优化的初稿，PRD-Writer 即可在三分钟内输出包含 **7 大标准章节** 的完整 PRD，涵盖功能描述、用户故事、优先级判定与验收标准。

### 核心优势

- 🌀 **Anthropic 三阶段工作流** — Context Gathering → Refinement & Structure → Reader Testing，AI 与用户形成清晰高效的协作循环
- 📊 **MoSCoW 优先级框架** — P0/P1/P2/Won't 精确标注每个功能优先级，告别模糊的优先级描述
- ✅ **Given/When/Then 验收标准** — BDD 规范格式，开发团队可直接转化为自动化测试用例
- 🌐 **中英双语输出** — 自动适配输出语言，轻松应对出海产品与国际团队协作
- 📋 **三种灵活输入模式** — 支持从零描述、旧稿优化、飞书文档链接三种方式启动
- 🔗 **飞书深度集成** — 支持直接读取飞书云文档/知识库，自动解析结构化内容
- 🗂️ **7 章节标准结构** — 覆盖产品需求文档全部核心要素，输出即可直接使用

## 🎯 功能特性

### 三阶段科学工作流

PRD-Writer 采用 Anthropic 推荐的三阶段协作范式，确保 AI 与用户之间形成清晰、高效的协作循环：

| 阶段 | 名称 | 核心动作 |
|------|------|---------|
| 第一阶段 | **Context Gathering** 需求收集 | AI 主动提问，澄清产品背景、目标用户、核心场景、技术约束 |
| 第二阶段 | **Refinement & Structure** 结构化生成 | 基于收集的信息，一次输出完整 7 章节 PRD |
| 第三阶段 | **Reader Testing** 验收反馈 | 从读者视角审查文档完整性，确认关键判断点，迭代更新 |

### MoSCoW 优先级框架

告别模糊的优先级描述。PRD-Writer 内置业界标准的 MoSCoW 优先级框架，为每个功能点标注明确优先级：

| 标识 | 含义 | 说明 |
|------|------|------|
| **P0 / Must** | 必须有 | 核心功能，上线门槛，缺失则系统无法正常使用 |
| **P1 / Should** | 应该有 | 重要功能，显著提升产品价值，不影响主流程 |
| **P2 / Could** | 可以有 | 增强功能，体验提升，本次可选 |
| **Won't** | 这次不做 | 明确排除，列入未来规划 |

### Given/When/Then 验收标准格式

每个 P0 和 P1 功能均配套输出符合 **Behavior-Driven Development（BDD）** 规范的验收标准：

```
Given（前提条件）+ When（触发动作）+ Then（预期结果）
```

开发团队可直接将此验收标准转化为自动化测试用例，实现「需求即测试」。

### 7 章节标准 PRD 结构

每份生成的 PRD 均包含以下标准章节：

| # | 章节 | 说明 |
|---|------|------|
| 1 | **产品概述** | 背景、目标、范围、价值主张 |
| 2 | **用户分析** | 用户画像、用户旅程 |
| 3 | **功能需求** | MoSCoW 优先级功能清单 + 功能详述 |
| 4 | **成功指标** | 北极星指标、领先指标 / 滞后指标 |
| 5 | **验收标准** | Given/When/Then 格式验收用例 |
| 6 | **技术约束** | 技术栈、集成依赖、安全合规 |
| 7 | **附录** | 术语表、版本记录、联系人 |

> "验收标准再加一个异常流程"

## 🔄 三阶段工作流详解

### 阶段一：Context Gathering（需求收集）

AI 通过提问收集关键信息，用于填充 PRD 各章节。

**收集维度**：

| 维度 | 对应章节 | 典型问题示例 |
|------|---------|------------|
| 项目背景与目标 | 产品概述 | 这个产品解决什么问题？目标是什么？ |
| 目标用户 | 用户分析 | 谁会用这个功能？他们的痛点是什么？ |
| 核心功能 | 功能需求 | 最核心的 3 个功能是什么？ |
| 优先级判断 | 功能需求 | 哪些必须有？哪些可以放到下一版？ |
| 成功指标 | 成功指标 | 希望达成什么可量化的结果？ |
| 验收标准 | 验收标准 | 怎样算「做好了」？有哪些关键场景？ |
| 技术约束 | 技术约束 | 有没有技术限制？依赖哪些系统？ |

**提问策略**：
- 每次提出 2-3 个问题，避免一次问太多
- 根据输入模式调整提问范围（模式 A 全量提问，模式 B/C 只问缺口）

**阶段输出**：一份「需求收集清单」供用户确认，无误后进入下一阶段。

---

### 阶段二：Refinement & Structure（结构化生成）

基于确认的需求，使用模板生成完整 PRD。

**生成顺序**：

| 顺序 | 章节 | 优先级 | 说明 |
|------|------|--------|------|
| 1 | 产品概述 | P0 必须 | 背景、目标、范围、价值主张 |
| 2 | 用户分析 | P0 必须 | 用户画像、用户旅程 |
| 3 | 功能需求 | P0 必须 | MoSCoW 优先级功能清单 + 功能详述 |
| 4 | 成功指标 | P1 应该 | 北极星指标、领先 / 滞后指标 |
| 5 | 验收标准 | P1 应该 | Given/When/Then 格式 |
| 6 | 技术约束 | P2 可以有 | 技术栈、集成依赖、安全合规 |
| 7 | 附录 | P2 可以有 | 术语表、版本记录、联系人 |

**生成规则**：
- 用户未提供的信息用 `[待确认]` 占位，并标注 `⚠️ 待补充`
- MoSCoW 优先级必须体现在功能清单中
- 验收标准必须使用 Given/When/Then 格式

---

### 阶段三：Reader Testing（验收反馈）

引导用户检查 PRD 完整性，确认关键判断点，迭代更新。

**检查清单**：

- [ ] **产品概述**：背景描述是否符合预期？目标是否可量化？
- [ ] **用户分析**：目标用户是否覆盖全面？用户旅程是否完整？
- [ ] **功能需求**：P0/P1/P2 分级是否符合预期？是否有遗漏的核心场景？
- [ ] **成功指标**：指标目标值是否合理？领先指标和滞后指标逻辑关系是否成立？
- [ ] **验收标准**：Given/When/Then 格式是否清晰？主要异常流程是否覆盖？
- [ ] **技术约束**：技术栈和集成依赖是否准确？

**关键判断点确认**：

> "『首次响应时长』目标值设为 24 小时，你希望更激进（4 小时）还是更保守（48 小时）？"
>
> "如果只给你 2 周开发时间，你会保留哪 3 个功能？"

## 📂 三种输入模式

### 模式 A：需求描述 → 完整 PRD

**适用场景**：产品从 0 到 1，或有新功能想法但尚未形成文档。

```
你：我想做一个让用户组队打卡运动的应用

PRD-Writer：
好的，我来帮你生成 PRD。需要确认几个关键信息：

1. 目标用户是 C 端普通用户，还是 B 端健身机构？
2. 组队打卡的核心激励机制是什么？（排名？奖励？社交？）
3. 需要记录哪些运动数据？（步数？时长？卡路里？）

请告诉我，我会生成完整的 7 章节 PRD。
```

---

### 模式 B：待优化 PRD → 完善 PRD

**适用场景**：已有初稿或框架性文档，但结构混乱、内容残缺、优先级不清晰。

```
你：[粘贴了一段 300 字的需求描述]

PRD-Writer：
已读取你的需求描述，核心功能已清晰。
以下章节需要补充：

- ❌ 验收标准：缺少具体验收条件
- ⚠️ 成功指标：只提到「提高效率」，没有量化指标
- ⚠️ 技术约束：没有提及技术限制或外部依赖

请提供：
1. 核心功能必须覆盖的验收场景（3-5 个）
2. 你希望通过这个功能达成什么可量化的结果？
3. 有没有技术限制或外部依赖？

我会基于这些信息生成完整 PRD。
```

---

### 模式 C：需求文档链接 → 多轮 PRD

**适用场景**：需求分散在飞书云文档或知识库中，需要多轮协作迭代。

```
你：请基于这个飞书文档帮我生成 PRD
    https://xxx.feishu.cn/doc/xxxxx

PRD-Writer：
已读取你的飞书文档，涵盖主要功能模块。

1. 目标用户是否包括客服侧？（文档中未提及客服操作需求）
2. 成功指标中「处理时长」目标值是多少？
3. 是否需要移动端支持？

请告诉我以上信息，我将生成完整 PRD。
```

## ❓ 常见问题 FAQ

### Q1: PRD-Writer 和普通的文档生成有什么不同？

PRD-Writer 不只是生成文档——它通过三阶段工作流确保文档的**完整性和可操作性**。MoSCoW 优先级让团队明确先做什么，Given/When/Then 验收标准让开发团队直接对照执行。普通生成工具往往输出的文档缺少优先级和验收标准，无法直接用于开发。

### Q2: 已有初稿可以直接上传吗？

可以。使用**模式 B**粘贴你的初稿，PRD-Writer 会分析文档结构，识别缺失章节，针对缺口进行定向补充，无需从头开始。

### Q3: 支持哪些语言的输出？

支持**中文**和**英文**两种输出语言。PRD-Writer 会根据你输入的语言自动判断输出语言，也可以明确指定"请用英文输出"。

### Q4: 飞书文档支持哪些类型？

支持飞书**云文档**（docx）和**知识库**（wiki）两种链接格式。AI 会自动读取文档内容并解析结构化信息。

### Q5: 生成的验收标准可以直接用于测试吗？

Given/When/Then 格式的验收标准设计初衷就是让开发团队和 QA 可以直接将其转化为自动化测试用例。每个验收用例包含明确的前提条件、触发动作和可验证的预期结果，无需额外解读。

### Q6: 如何获得更详细的操作指南？

查看飞书云文档中的详细使用指南：

- 📖 **[PRD-Writer 使用指南](https://my.feishu.cn/wiki/Qd3qw5Xn7i3xG6k6llkc2ezYnYd)** — 完整操作说明与最佳实践
- 📄 **[PRD 模板参考](https://my.feishu.cn/wiki/NHOLwynpoiHFrMk3j4IcDFoCnOh)** — 中英文 PRD 标准模板
- 🔧 **[开发说明文档](https://my.feishu.cn/wiki/Np8jwlGagivfUDk9rRuc75XrnQD)** — 技能开发细节与架构说明

### Q7: 如果生成的内容不符合预期怎么办？

进入**阶段三（Reader Testing）**后，你可以针对任意章节提出修改要求：
> "第三章的功能优先级需要调整"  
> "验收标准再加一个异常流程处理"  
> "成功指标的目标值改为..."

PRD-Writer 支持局部更新，并记录版本变更历史。
