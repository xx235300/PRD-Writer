# PRD 模板（中文标准版）

> **版本**：v2.0 标准版
> **适用场景**：AI 驱动的产品需求文档生成
> **章节结构**：七章标准结构（项目概述 → 用户分析 → 需求详述 → 成功指标 → 验收标准 → 技术约束 → 附录）
> **优先级体系**：MoSCoW 四级分类（Must / Should / Could / Won't），映射 P0 ~ P2
> **验收标准格式**：Given / When / Then（行为驱动开发格式）
> **指标体系**：Leading 领先指标 + Lagging 滞后指标双轨体系
> **字数要求**：每个章节包含章节说明、填写说明、模板正文、示例内容四部分，整体不少于 15,000 字

---

## 第一章、项目概述

### 章节说明

本章用于描述项目的背景、目标、范围和核心价值，让读者在 5 分钟内理解「这个产品是什么」「为什么要做」以及「做到什么程度才算成功」。本章是整份 PRD 的基础，后续所有章节的内容都必须与本章定义的目标和范围保持一致。项目启动之初即应填写，经团队评审对齐后作为共识基线。在项目执行过程中，如目标和范围发生调整，必须同步更新本章并通知所有相关方。

### 填写说明

- **背景与问题**：描述当前业务中存在的主要痛点，具体到「什么人」「在什么场景下」「遇到了什么问题」，用数据说明问题的严重程度和影响范围。避免空泛的「用户体验不好」「效率低下」等表述。
- **项目目标**：用 SMART 原则（具体 / 可衡量 / 可达成 / 相关 / 有时限）定义 3~5 个核心目标，每个目标尽量附带量化指标、目标值和衡量方式。没有数据的项目可以先给定性目标，待上线后补充数据。
- **范围定义——包含**：明确列出本次项目要做的工作内容，按功能模块分类。不要在此处写入实现细节（如数据库表设计），只写用户可感知的功能范围。
- **范围定义——不包含**：明确列出本次项目不做但容易被误以为要做的事项，以及将来可能做但本次不做的事项。写清楚排除原因，防止范围蔓延（Scope Creep）。
- **核心价值主张**：用一句话（不超过 50 字）回答「用户为什么需要这个功能，它给用户带来了什么核心价值」。通常是一个爽点（Pain Relief）或收益点（Gain）。

### 示例内容

```markdown
# 在线教育平台课程反馈系统

## 1.1 背景与问题

当前平台的用户反馈渠道极度分散：用户通过微信群、App 应用商店评价、客服聊天工具、班级社群等多个渠道提交关于课程质量的反馈。产品团队每周收到约 120 条原始反馈，其中来自应用商店差评的占 38%，来自班级社群截图转发的占 27%，来自客服转交的占 22%，仅有 13% 来自官方反馈入口。

这种分散状况造成三个核心痛点：

**痛点 1：反馈处理效率极低。** 客服团队每天花费 2.5 小时将反馈人工分类到对应的产品模块（视频、音频、PPT、作业等），平均每条反馈耗时 8 分钟。由于缺乏统一格式，信息经常缺失，客服还需要反复联系用户补充细节。客服负责人统计，当前反馈从用户提交到首次响应的平均时长为 52 小时，远超用户可接受的等待阈值（4 小时）。

**痛点 2：重要问题系统性遗漏。** 2025 年第四季度，有 3 次课程视频大规模损坏的事故，用户通过应用商店差评和微信群反馈，但产品团队在 3~5 天后才从客服周报中得知，错过了最佳响应窗口。这 3 次事故在社交媒体上引发的客诉造成了约 18 万元的直接退费损失。

**痛点 3：用户感知「反馈石沉大海」。** 产品团队对 2025 年 12 月的 1,200 条反馈样本分析发现，67% 的反馈在提交后用户再无任何消息通知，用户只能通过猜测判断反馈是否被看到。同期 App 应用商店中包含「反馈没回音」「说了没人理」等表述的差评占总差评量的 31%。

**当前痛点**：
- 反馈分散在 5+ 个渠道，产品团队无法获取完整的一手数据
- 客服人工分类平均错误率达 35%，导致反馈经常被转给错误的负责人
- 重要问题平均延迟 3.8 天才被产品团队知晓，错过最佳处理窗口
- 用户对「反馈无响应」的投诉占应用商店差评总量的 31%

## 1.2 项目目标

| 目标编号 | 目标描述 | 衡量指标 | 目标值 | 数据来源 |
|----------|----------|----------|--------|----------|
| OBJ-001 | 建立统一的反馈收集入口，提升反馈提交量 | 日均反馈提交量 | 较上线前提升 30%，达到日均 156 条 | 埋点数据 |
| OBJ-002 | 通过自动分类减少人工处理工作量 | 客服每条反馈平均处理时长 | 从 15 分钟降至 8 分钟 | 客服系统记录 |
| OBJ-003 | 缩短用户反馈的首次响应时长 | 首次响应时长（P50） | 从 52 小时缩短至 24 小时以内 | 反馈系统后台 |
| OBJ-004 | 减少因「反馈无响应」引发的应用商店差评 | 应用商店差评中「反馈无响应」类占比 | 从 31% 降至 15% 以下 | App Store Connect / Google Play Console |
| OBJ-005 | 产品团队能及时获知重要问题 | 重要问题（影响 > 10% 用户的故障）平均发现时长 | 从 3.8 天缩短至 4 小时以内 | 内部故障记录系统 |

## 1.3 范围定义

### 1.3.1 包含范围

- **PC Web 端反馈提交入口**：课程详情页右侧固定「意见反馈」按钮，点击弹出反馈表单（支持问题类型选择、文字描述、图片上传）
- **移动端 H5 反馈页面**：独立反馈提交页面，可在微信和小程序外浏览器中打开
- **用户端历史反馈页面**：用户可查看自己提交的所有反馈记录及当前处理状态
- **客服后台反馈管理模块**：反馈列表（支持多条件筛选、关键词搜索）、反馈详情查看、状态变更、内部备注、导出功能
- **自动归类引擎**：基于关键词规则（第一阶段）+ 课后反馈类型预标注，自动将反馈分配至对应课程模块负责人
- **统计面板**：面向产品团队，展示反馈量趋势、类型分布、平均响应时长、满意度评分等核心指标

### 1.3.2 不包含范围

- **移动端 App 内嵌反馈入口**（Native/Hybrid）：本次仅提供 H5 页面，App 内嵌反馈功能在下一版本迭代。App 团队表示嵌入成本较高，需单独排期（约 8 人天）。
- **用户之间的反馈互动社区**：即用户可以在反馈下互相回复、点赞的功能，排除在外。原因：社区运营复杂度高，超出本期项目范围。
- **与 Zendesk / Salesforce / 逸创等第三方工单系统的深度集成**：本期仅支持内部处理，第三方集成在 v2.0 讨论。原因：第三方 API 对接测试周期较长。
- **基于大模型的智能语义归类和自动回复功能**：本期使用规则引擎（关键词 + 类型预判），LLM 能力在 v2.0 评估。原因：LLM API 成本和响应延迟需进一步评估。
- **实时客服聊天（即时通讯）功能**：本期仅支持异步反馈提交和处理通知，不包含实时 IM 能力。
- **匿名反馈**：本期必须登录后才能提交反馈，便于对重复反馈去重和追溯。匿名反馈在 v2.0 评估。

## 1.4 核心价值主张

让平台用户遇到课程问题时能一键提交反馈并实时追踪处理进展，让产品团队首次拥有完整的用户声音数据，从而系统性减少「反馈无响应」导致的用户流失和应用商店差评。
```

## 第二章、用户分析

### 章节说明

本章深入分析目标用户群体，理解他们的角色、目标、行为模式和痛点，为后续的需求详述和验收标准设计提供依据。本章应该在「需求详述」之前完成，帮助整个团队站在用户视角思考，而不是站在功能实现视角。本章定义的用户画像将直接作为「用户故事（User Story）」的输入，每一条用户故事都应能回溯到本章定义的一个或多个用户角色。

### 填写说明

- **目标用户画像**：为每个用户角色建立完整的画像卡片，包含角色定义、用户背景、日常行为、核心目标（Goal）、关键痛点（Pain）和使用频率。使用「峰终定律」思考用户旅程中的关键时刻（Peak Moment）和结束时刻（End Point）。
- **用户旅程与需求**：从用户视角梳理从「发现需求」到「任务完成」的完整旅程，划分 4~6 个阶段，分析每个阶段的用户行为、触点、需求和痛点。重点识别那些让用户流失或放弃的关键阻力点。
- **用户故事（User Story）**：使用标准格式「作为 [角色]，我希望 [功能]，以便 [收益]」编写用户故事。每个核心功能对应 1~3 条用户故事。用户故事是需求详述章节的直接输入。
- **术语定义预备**：在开始填写本章前，先梳理本章中出现的所有术语，给出精确定义，确保团队对同一概念的理解一致。术语表在附录中统一维护。

### 示例内容

```markdown
## 2.1 目标用户画像

### 2.1.1 主要用户

| 属性 | 内容 |
|------|------|
| 角色 | 平台付费学员 |
| 用户背景 | 每日或每周登录平台观看课程视频，完成课后作业，已付费 499~2999 元购买课程，对课程质量有明确预期 |
| 核心目标 | 快速提交遇到的课程问题并在合理时间内得到响应，继续正常学习 |
| 关键痛点 | 不知道在哪里提交反馈，邮件提交后完全没有回音，无法查看反馈处理进度，不知道问题是被忽略了还是正在处理 |
| 使用频率 | 遇到问题时每月 1~2 次，属于低频但高价值的行为 |
| 技术能力 | 中等，熟悉主流 App 和 Web 产品的交互模式，能独立完成表单填写 |

### 2.1.2 次要用户

| 属性 | 内容 |
|------|------|
| 角色 | 一线客服专员 |
| 用户背景 | 负责接收和处理来自各渠道（微信、App 评价、客服 IM、邮件）的用户反馈，每日处理约 40~60 条反馈工单 |
| 核心目标 | 高效分类和转派反馈工单，减少重复劳动，确保重要问题不被遗漏，在 SLA 时限内完成响应 |
| 关键痛点 | 各渠道反馈格式不统一，人工分类耗时（每条 8 分钟），转派时需要反复确认负责人，不知道用户是否已通过其他渠道重复提交 |
| 使用频率 | 每个工作日多次，反馈管理系统是其日常工作的核心工具 |
| 技术能力 | 高，能熟练使用内部工单系统，对后台产品有基本认知 |

| 属性 | 内容 |
|------|------|
| 角色 | 产品经理 |
| 用户背景 | 负责课程产品质量，关注用户反馈数据的趋势和洞察，推动产品改进项目落地 |
| 核心目标 | 及时获取用户对课程的真实反馈，将高频问题转化为产品迭代需求，向管理层展示反馈处理效率的提升 |
| 关键痛点 | 当前没有统一的反馈数据视图，只能从客服周报中获取二手信息，数据滞后 3~5 天，无法进行实时分析 |
| 使用频率 | 每周 2~3 次，查看数据面板和重点反馈详情 |
| 技术能力 | 高，熟练使用数据分析工具 |

## 2.2 用户旅程与需求

### 2.2.1 用户主流程

- **阶段一【发现】**：用户在学习过程中发现课程存在质量问题（视频无法播放、PPT 缺少关键页、作业题目有误等）
- **阶段二【评估】**：用户在「是否值得反馈」和「反馈的预期收益」之间权衡，典型心理活动是「这问题很多人遇到吗？反馈有用吗？会不会浪费时间？」
- **阶段三【行动】**：用户找到反馈入口，填写问题描述，提交反馈
- **阶段四【等待】**：用户在等待过程中产生焦虑，不确定反馈是否被收到，不知道要等多久
- **阶段五【接收】**：用户收到处理结果通知（处理完成 / 已转派等），对处理结果做出满意度评价

### 2.2.2 各阶段触点与痛点分析

| 阶段 | 触点 | 用户行为 | 核心需求 | 当前痛点 | 设计机会 |
|------|------|----------|----------|----------|----------|
| 发现 | 课程视频播放页 | 用户发现视频卡顿、无法加载、声音画面不同步 | 希望立刻知道这是个案还是普遍问题 | 无法判断是自己的网络问题还是课程本身的问题 | 提供「课程故障自检」能力，让用户一键确认是否为平台故障 |
| 评估 | App 评价页 / 微信群 | 用户看到其他学员的差评，或在班级微信群看到同学抱怨同类问题 | 确认这不是个别现象，值得正式提交反馈 | 反馈渠道不明确，不知道该在哪里提交正式反馈 | 在课程页面提供明显的反馈入口，降低提交门槛 |
```

```markdown
## 3.2 核心功能详述

### 3.2.1 反馈表单（P0）

#### 功能概述

用户在使用平台过程中遇到课程质量问题时，可通过反馈表单快速提交问题描述和相关截图。系统自动归类问题并分配给对应课程模块负责人处理，用户提交后立即获得反馈编号，可在历史记录中实时追踪处理进度。本功能对应用户故事 US-001、US-002、US-004。

#### 业务规则

- **问题类型必选**：用户提交反馈前必须从预设类型中选择一项，未选择时「提交」按钮禁用。问题类型共 5 类，每类下有子类型：
  - 视频异常（视频无法播放 / 画面卡顿 / 声音画面不同步 / 画质模糊）
  - 音频异常（无声音 / 噪音 / 与视频不同步）
  - 文档异常（PPT 缺少页 / PDF 无法打开 / 文档内容错误）
  - 作业问题（题目有误 / 无法提交 / 答案错误）
  - 其他问题（以上类型都无法覆盖的问题）
- **文字描述必填**：描述文字最少 10 个字符，最多 2000 个字符，输入框实时显示已输入字符数。
- **图片上传可选**：最多上传 3 张图片，每张不超过 5MB，支持 jpg / png / gif 格式，不支持其他文件类型。第 4 张及以上的上传操作被阻止并显示提示「最多上传 3 张图片」。
- **反馈编号生成规则**：提交成功后生成唯一反馈编号，格式为 FB-YYYYMMDD-XXX（如 FB-20260328-001）。反馈编号在反馈详情页显示并可复制，用于用户查询。
- **自动预标注**：根据用户当前所在课程页面，自动填充课程名称和章节名称到反馈表单，减少用户输入成本。
- **提交频率限制**：同一用户在 1 分钟内最多提交 3 条反馈，超过后提示「您的提交频率过高，请稍后再试」。

#### 前置条件

- 用户已登录账号
- 用户在产品使用中（非游客状态）

#### 主流程

1. 用户点击课程页面右侧「意见反馈」悬浮按钮（或在帮助中心点击「提交反馈」入口）
2. 系统弹出反馈表单弹窗，同时记录反馈来源页面（source 参数）
3. 系统根据用户当前页面自动填充课程名称（不可编辑，供用户确认）
4. 用户从 5 大类型中选择问题类型（如当前问题不在预设类型中选择「其他问题」）
5. 用户在文字描述框中详细描述遇到的问题（系统实时显示已输入字符数）
6. 用户可选点击「添加图片」上传截图（最多 3 张，支持点击选择或拖拽上传）
7. 用户点击「提交反馈」按钮
8. 系统校验必填项，如有缺失在对应字段下方显示红色提示，不提交
9. 系统提交反馈数据到后端，成功后显示「提交成功」弹窗，告知反馈编号和预期处理时长（24 小时内）
10. 弹窗在 3 秒后自动关闭，或用户点击「查看详情」跳转到历史反馈详情页

#### 异常流程

- **网络异常**：提交时网络断开，系统显示「提交失败，请检查网络后重试」红色提示，用户填写的内容完整保留，输入框不重置
- **提交内容包含敏感词**：系统显示「反馈内容包含不支持的表述，请修改后重试」，不透露具体敏感词，用户内容保留
- **图片上传失败**（单张失败，不影响其他图片和表单提交）：失败图片显示红色叹号图标和「上传失败」文字，用户可点击重试或删除
- **重复提交拦截**：用户在 1 分钟内第 4 次点击提交时，显示「您的提交频率过高，请稍后再试」提示，拦截提交

### 3.2.2 历史反馈列表（P0）

#### 功能概述

用户提交反馈后，可进入「我的反馈」页面查看自己提交的所有反馈记录列表和每条反馈的当前处理状态。反馈列表默认按提交时间倒序排列，用户可通过状态标签筛选不同处理阶段的反馈。本功能对应用户故事 US-003。

#### 业务规则

- **列表排序规则**：默认按「提交时间」倒序排列（最新提交的在前），用户可切换为「按更新时间倒序」
- **状态筛选**：支持按「全部 / 待处理 / 处理中 / 已完成 / 已关闭」5 种状态筛选，默认显示「全部」
- **反馈卡片信息**：每张卡片显示问题类型标签（带颜色区分）、描述前 60 字（超出 60 字显示省略号）、提交时间（相对时间格式，如「3 小时前」「2 天前」）、处理状态标签
- **分页规则**：列表每页显示 20 条，采用下拉加载更多（Infinite Scroll），不再使用传统分页控件
- **空状态**：用户从未提交过反馈时，显示空状态插画（课程相关的卡通插图）和文案「还没有反馈记录，快去提交第一个问题吧」，下方显示「去反馈」按钮，点击后跳转到最近访问的课程页面
- **数据来源**：列表数据来自用户提交的历史反馈，与客服后台共用同一数据源，用户只能看到自己的反馈

#### 前置条件

用户已登录账号

#### 主流程

1. 用户从「我的」→「我的反馈」入口进入历史反馈列表页
2. 系统默认加载全部状态的反馈列表，按提交时间倒序
3. 用户可在顶部筛选栏切换状态标签查看特定状态的反馈
4. 用户下拉到底部时系统自动加载下一页（20 条 / 页）
5. 用户点击任意一条反馈卡片，进入该反馈的详情页

#### 异常流程

- **网络异常导致列表加载失败**：页面中央显示加载失败提示「加载失败，请检查网络后重试」，下方显示「重新加载」按钮

### 3.2.3 客服后台反馈列表（P0）

#### 功能概述
```

## 第四章、成功指标

### 章节说明

本章定义衡量产品是否成功的量化指标体系。指标体系采用「北极星指标 + 双轨指标」结构：北极星指标是最能反映产品核心价值的单一指标，是所有团队成员共同追求的目标；领先指标（Leading Indicators）是可以高频观测的短期行为信号，是北极星指标的前置预测信号，用于团队在北极星指标变化之前就及时发现问题并调整策略；滞后指标（Lagging Indicators）是最终的业务结果，与公司核心商业目标挂钩，通常需要较长时间才能显现。领先指标与滞后指标之间存在因果链条，理解这种因果关系是团队建立数据驱动决策习惯的基础。本章的所有指标数据来源应与第三章的数据埋点需求一一对应，确保上线后指标可测量。

### 填写说明

- **北极星指标**：选择 1 个最能反映本产品价值的单一指标。标准是：这个指标变好了，是否能确认产品就成功了？避免选择虚荣指标（如总注册用户数）或过于间接的指标（如 NPS）。每个北极星指标需要附带定义、目标值和测量频率。
- **领先指标（Leading Indicators）**：选择 3~6 个可以在短期内（1~4 周内）观测到变化的行为指标，这些指标是北极星指标的前置信号。领先指标的理想衡量周期是「每周观测一次」，不追求每天都变化，但需要保持对趋势的敏感度。
- **滞后指标（Lagging Indicators）**：选择 2~4 个最终反映业务价值的指标，这些指标通常与公司核心 KPI 挂钩（如收入、留存、满意度）。滞后指标的变化可能需要 1~3 个月才能显现，团队应保持耐心，避免因为短期未见变化就判定产品失败。
- **指标关系说明**：描述领先指标与滞后指标之间的因果逻辑，帮助团队理解「为什么要关注短期行为指标」。这是让管理层和非数据背景的团队成员理解产品工作价值的关键。
- **技术性能指标**：非业务层面的系统质量指标，用于确保产品「不仅业务成功，技术上也稳健可靠」。这类指标面向 SRE 和开发团队，通常不面向产品经理但产品经理需要了解其基本要求。

```markdown
## 4.1 北极星指标

反馈系统上线后，用户反馈从提交到首次响应的平均时长（Mean First Response Time）。

| 指标名称 | 定义 | 目标值 | 测量频率 |
|----------|------|--------|----------|
| 平均首次响应时长 | 从用户提交反馈（timestamp_submitted）到客服首次回复（timestamp_first_response）的时间间隔，按自然日统计每日均值 | 上线第 1 个月内：≤ 24 小时；上线第 3 个月后：≤ 8 小时 | 每日统计，每周三管理层数据例会复盘 |

## 4.4 指标关系说明

反馈提交率提升（领先指标）意味着更多用户愿意通过产品内渠道而非应用商店差评来表达不满。如果用户连反馈都懒得提，那往往意味着他们选择直接离开甚至差评，而不是给你改进的机会。因此，高反馈提交率是一个健康的信号，说明用户对平台仍有期待并愿意参与改进。

反馈提交至状态变更的转化率（领先指标）反映用户对反馈处理的关注度。如果用户提交反馈后持续查看详情页，说明他们在等待结果，这是产品团队必须及时响应以维持用户信任的关键窗口。如果这一转化率下降（用户不再追踪），可能意味着用户已经放弃了对平台响应能力的信任，这是滞后指标（满意度评分）恶化的前兆。

首次响应时长缩短（领先指标）直接影响用户满意度和应用商店评分（滞后指标）。心理学上的「峰终定律」表明，用户对一次体验的评价主要由最高峰和结束时的感受决定。对于反馈体验来说，首次响应就是「最高峰」——如果用户在数小时内就得到响应，即便最终结果不如预期，用户的整体感受也会较好。

反馈处理满意度评分（滞后指标）和客服工单下降率（滞后指标）是最终反映反馈系统是否真正解决用户问题的指标。如果满意度评分持续低迷，说明反馈处理流程本身（不只是速度，还包括处理质量和用户沟通方式）需要优化。客服工单下降则是「反馈系统减少了无效沟通」的直接证明。

| 系统可用性 | 系统正常服务时间占总时间的比例（排除计划内维护窗口） | ≥ 99.9%（月均停机时间 < 44 分钟）|
| 反馈提交接口吞吐量 | 反馈提交接口每秒可处理的提交请求数（QPS） | 支持 100 条 / 分钟的峰值提交量（基于 OBJ-001 目标反推）|
| 故障恢复时间（MTTR） | 从系统故障发生到服务恢复正常的中位时间 | ≤ 30 分钟 |
```

## 第五章、验收标准

### 章节说明

本章使用 Given / When / Then 格式（行为驱动开发格式）定义功能的验收标准（Acceptance Criteria），每一条验收标准都必须包含 Given（前提条件）、When（触发操作）和 Then（预期结果）三个要素。Given / When / Then 格式的核心价值在于：它迫使产品经理将「前提条件」「具体操作」和「可验证的结果」三者都写清楚，避免使用模糊的表述（如「系统反应正常」「用户体验良好」）。开发工程师可以通过验收标准直接理解功能边界，测试工程师可以将每一条验收标准直接转化为一条测试用例，产品经理可以通过验收标准判断功能是否达到上线标准。本章应覆盖每个 P0 功能的正常流程（Happy Path）和主要异常流程（Edge Cases）。

### 填写说明

- **Given 部分**：描述测试执行前系统应处于的状态，包括用户登录状态、数据准备、网络环境等前提条件。Given 应该是可复现的、可在测试环境中设置的。
- **When 部分**：精确描述用户或系统执行的操作，包括操作方式（点击 / 输入 / 拖拽等）、操作目标（哪个按钮、哪个输入框）和操作参数（输入什么值）。
- **Then 部分**：描述操作后系统应该产生的可观测变化，包括界面变化（显示什么文字 / 什么弹窗）、数据变化（数据库中记录了什么）、通知变化（发送了什么通知）。所有结果应该是可以自动化验证的。
- **覆盖范围**：每个 P0 功能至少 5 条验收标准（3 条正常流程 + 2 条异常流程），P1 功能至少 3 条。不写 Won't 级别的功能验收标准。
- **边界值测试**：对于数值类限制（如字符数限制、图片数量限制），必须包含边界值测试用例（最小值 - 1、精确值、最大值、最大值 + 1）。
- **编号规范**：使用 AC（Acceptance Criteria）编号，如 AC-001、AC-002，便于与测试用例编号对应。

### 模板正文

```markdown
## 5.1 功能验收标准

### 5.1.1 【功能名称 A】

**验收标准 N：【场景名称】**

- **Given**：【前提条件描述】
- **When**：【触发操作描述】
- **Then**：【预期结果描述】

**验收标准 N+1：【场景名称】**

- **Given**：【前提条件描述】
- **When**：【触发操作描述】
- **Then**：【预期结果描述】

### 5.1.2 【功能名称 B】

【同上格式，每个 P0 功能至少 5 条验收标准】
```

### 示例内容

```markdown
## 5.1 功能验收标准

### 5.1.1 反馈表单提交

**验收标准 AC-001：正常提交反馈（完整流程）**

- **Given**：用户已登录账号，正在课程 A 的视频播放页面，视频无法正常播放
- **When**：用户点击页面右侧「意见反馈」悬浮按钮 → 系统弹出反馈表单 → 用户确认课程名称为「课程 A」→ 用户选择问题类型为「视频异常」→「视频无法播放」→ 用户在描述框中输入「播放到第 12 分钟时提示网络错误，重试多次无效」（50 个字符）→ 用户点击「添加图片」上传 1 张截图（2MB，jpg 格式）→ 用户点击「提交反馈」按钮
- **Then**：系统显示「提交成功」弹窗，弹窗内容包含反馈编号「FB-20260328-001」和文案「已收到您的反馈，我们将在 24 小时内处理」，弹窗在 3 秒后自动关闭，页面恢复无弹窗状态

**验收标准 AC-002：必填项校验——问题类型未选**

- **Given**：用户已登录账号，点击「意见反馈」悬浮按钮，反馈表单已弹出
- **When**：用户未选择任何问题类型，直接在描述框中输入「视频无法播放」并点击「提交反馈」按钮
- **Then**：系统不提交表单，在问题类型选择区域下方显示红色提示文字「请选择问题类型」（10 号字体），「提交反馈」按钮保持可点击状态但点击后再次触发相同校验

**验收标准 AC-003：必填项校验——描述文字不足**

- **Given**：用户已登录账号，点击「意见反馈」悬浮按钮，反馈表单已弹出
- **When**：用户选择问题类型为「视频异常」→「视频无法播放」，在描述框中输入 8 个字符「播放报错」，点击「提交反馈」按钮
- **Then**：系统不提交表单，在描述框下方显示红色提示文字「请详细描述问题，至少输入 10 个字符」，已输入的 8 个字符完整保留，用户可继续补全

**验收标准 AC-004：文本输入边界值——正好 10 个字符（最小有效值）**

- **Given**：用户已登录账号，点击「意见反馈」悬浮按钮，反馈表单已弹出
- **When**：用户选择问题类型为「其他问题」，在描述框中输入恰好 10 个字符「无法加载内容」，点击「提交反馈」按钮
- **Then**：系统校验通过，文本框下方不显示字数不足提示，系统正常提交，显示「提交成功」弹窗

**验收标准 AC-005：文本输入边界值——正好 2000 个字符（最大有效值）**

- **Given**：用户已登录账号，点击「意见反馈」悬浮按钮，反馈表单已弹出
- **When**：用户选择问题类型为「视频异常」，在描述框中输入恰好 2000 个字符（复制粘贴），字数统计显示「2000 / 2000」，点击「提交反馈」按钮
- **Then**：系统校验通过（字符数正好等于上限），正常提交，显示「提交成功」弹窗

**验收标准 AC-006：文本输入边界值——超出 2000 个字符（超限拦截）**

- **Given**：用户已登录账号，点击「意见反馈」悬浮按钮，反馈表单已弹出
- **When**：用户在文本框中粘贴超出 2000 个字符的内容（2050 个字符），字数统计显示「2050 / 2000」（数字变红）
- **Then**：系统阻止用户输入第 2001 个字符（即超出上限的内容无法粘贴进去），字数统计持续显示「2000 / 2000」，点击提交后正常提交（2000 字符以内有效）

**验收标准 AC-007：图片上传——第 1 张图片正常上传**

- **Given**：用户已登录账号，反馈表单已弹出，问题类型已选择，描述已填写
- **When**：用户点击「添加图片」按钮，在文件选择器中选择 1 张 2MB 的 jpg 图片，确认上传
- **Then**：上传区域显示该图片的缩略图（80×80px 白色背景灰色边框），缩略图右上角显示绿色勾选图标（表示上传成功），图片上传数量统计显示「1 / 3」

**验收标准 AC-008：图片上传——第 3 张图片正常上传**

- **Given**：用户已上传 2 张图片（均上传成功），反馈表单已弹出
- **When**：用户继续上传第 3 张图片（2MB，png 格式），确认上传
- **Then**：第 3 张缩略图正常显示，上传区域仍然显示「添加图片」按钮和虚线边框（说明还可以继续上传），图片数量统计显示「3 / 3」

**验收标准 AC-009：图片上传——超出 3 张被拦截**

- **Given**：用户已上传 3 张图片，反馈表单已弹出
- **When**：用户尝试上传第 4 张图片，确认上传
- **Then**：系统拦截第 4 张图片的上传请求，不显示该图片的缩略图，同时在上传区域下方弹出红色提示「最多上传 3 张图片，已达上限」，已上传的 3 张图片不受影响

**验收标准 AC-010：网络异常——提交失败且内容保留**

- **Given**：用户已登录账号，反馈表单已弹出，问题类型已选择，描述已填写（80 个字符），1 张图片已上传成功
- **When**：用户点击「提交反馈」按钮，同时使用 Charles Proxy 模拟网络断开（返回网络错误）
- **Then**：页面显示红色错误提示「提交失败，请检查网络后重试」（位于提交按钮下方），页面不跳转，用户已选择的问题类型、描述文字、已上传的图片均完整保留，用户可修复网络后重新点击提交

**验收标准 AC-011：反馈编号格式验证**

- **Given**：用户提交了一条反馈，提交成功
- **When**：用户在「提交成功」弹窗中查看反馈编号
- **Then**：反馈编号格式为 FB-YYYYMMDD-XXX（如 FB-20260328-001），其中 FB 为固定前缀，YYYYMMDD 为提交当天的日期，XXX 为当日该用户提交的第几条反馈（从 001 起，不足 3 位前补零）

**验收标准 AC-012：表单关闭——内容未保存时弹出确认**

- **Given**：用户已打开反馈表单，问题类型已选择，描述已填写（50 个字符），尚未提交
- **When**：用户点击弹窗右上角关闭按钮（X）
- **Then**：弹出确认对话框「是否放弃填写？您填写的内容将不会保存」，对话框包含「继续填写」（取消关闭）和「放弃」（确认关闭）两个按钮；点击「继续填写」关闭对话框，表单保持打开，内容完整保留；点击「放弃」关闭对话框，表单内容不保存

### 5.1.2 历史反馈列表

**验收标准 AC-013：列表加载——有数据时正常展示**

- **Given**：用户已登录账号，该用户历史有 5 条已提交的反馈记录（状态分别为：待处理 1 条、处理中 2 条、已完成 2 条）
- **When**：用户进入「我的反馈」页面
- **Then**：页面顶部显示「我的反馈」标题和总条数「共 5 条」，反馈列表按提交时间倒序显示 5 张卡片，每张卡片包含：问题类型标签（如「视频异常」，带颜色）、描述前 60 字 + 省略号（若有截断）、提交时间（显示「3 小时前」等相对时间）、处理状态标签（待处理：灰色标签 / 处理中：蓝色标签 / 已完成：绿色标签）

**验收标准 AC-014：列表加载——空状态展示**

- **Given**：用户已登录账号，该用户从未提交过任何反馈
- **When**：用户进入「我的反馈」页面
- **Then**：页面中央显示空状态插画（课程相关的简约卡通图，约 200×200px），插画下方显示文案「还没有反馈记录」，再下方显示「快去提交第一个问题吧」灰色副文案，最下方显示蓝色「去反馈」按钮

**验收标准 AC-015：状态筛选——按「已完成」筛选**

- **Given**：用户已登录账号，该用户有不同状态的反馈（待处理 3 条、处理中 2 条、已完成 5 条）
- **When**：用户点击筛选标签「已完成」
- **Then**：列表只显示状态为「已完成」的 5 条反馈，筛选标签「已完成」高亮显示（蓝色下划线或背景色），切换筛选后页面滚动到列表顶部

**验收标准 AC-016：反馈详情查看——信息完整性**

- **Given**：用户有 1 条状态为「处理中」的反馈，反馈编号 FB-20260328-001，问题类型「视频异常」→「视频无法播放」，描述为「播放到第 12 分钟时卡住，重试无效」（40 个字符），提交时间为今天上午 10:00，上传了 1 张截图
- **When**：用户在该反馈的卡片上点击，进入反馈详情页
- **Then**：详情页顶部显示反馈编号 FB-20260328-001（可点击复制）；显示问题类型「视频异常 > 视频无法播放」；显示完整描述文字（40 个字符，全文展示，无截断）；图片区域显示 1 张可点击放大的缩略图；处理记录时间线显示：上午 10:00【用户提交】（灰色）、下午 02:30【客服-张华标记为处理中】（蓝色）；当前处理状态显示「处理中」（蓝色标签）

**验收标准 AC-017：反馈详情——处理记录时间线升序展示**

- **Given**：用户有 1 条反馈，其处理记录包括 3 条（提交 → 标记处理中 → 已完成）
- **When**：用户进入该反馈的详情页，查看处理记录时间线
- **Then**：时间线从上到下按时间升序排列（最早的在上），每条记录包含时间戳（精确到分钟）、操作类型和操作人名称；当前最新状态在时间线底部高亮显示

### 5.1.3 客服后台反馈处理

**验收标准 AC-018：列表加载性能——10000 条数据在 2 秒内加载**

- **Given**：系统中已存在 10,000 条反馈数据（测试环境数据），客服人员已登录后台
- **When**：客服人员打开「反馈管理」列表页（不施加任何筛选条件，加载默认列表）
- **Then**：列表在 2 秒内完成加载并显示数据，前 20 条数据立即可见，下方显示骨架屏加载动画直到全部渲染完成

**验收标准 AC-019：状态变更——待处理 → 处理中**

- **Given**：客服人员正在查看一条「待处理」状态的反馈详情（反馈编号 FB-20260328-001）
- **When**：客服将状态选择器从「待处理」切换为「处理中」，点击「保存」按钮
- **Then**：页面顶部显示绿色成功提示「状态已更新为【处理中】」（2 秒后自动消失），反馈的状态标签立即更新为蓝色「处理中」，系统自动在处理记录时间线底部新增一条记录「【客服-张华】将状态更新为【处理中】」（包含操作时间，精确到分钟）

**验收标准 AC-020：状态变更——已完成状态不可回退**

- **Given**：客服人员正在查看一条「已完成」状态的反馈（反馈编号 FB-20260328-005）
- **When**：客服尝试将状态选择器切换为「待处理」或「处理中」，点击「保存」按钮
- **Then**：系统拦截操作，状态选择器不允许切换为「待处理」或「处理中」（已完成状态后选择器置灰或仅可切换为「已关闭」），页面不显示任何错误提示（交互上直接阻止，避免用户困惑）

**验收标准 AC-021：内部备注——添加后仅客服可见**

- **Given**：客服人员正在查看一条反馈详情（反馈编号 FB-20260328-003）
- **When**：客服在备注输入框中输入「已联系后端团队，确认是 CDN 故障，已修复」，点击「保存备注」
- **Then**：备注成功保存，在反馈详情页的「内部备注」区域显示该条备注（包含备注内容、添加时间和添加人昵称），用户端「我的反馈」详情页不显示该条备注（内部备注对用户完全不可见）

**验收标准 AC-022：搜索功能——按反馈编号精确搜索**

- **Given**：系统中存在多条反馈，其中反馈编号 FB-20260328-001 对应一条关于「视频无法播放」的反馈
- **When**：客服在反馈列表右上角的搜索框中输入「FB-20260328-001」，点击搜索按钮（或按回车键）
- **Then**：列表立即筛选出该编号对应的唯一一条反馈，其他反馈不显示；搜索结果中反馈编号「FB-20260328-001」高亮显示（黄色背景）；搜索框保持显示当前搜索关键词；提供「清除搜索」按钮，点击后恢复完整列表

**验收标准 AC-023：搜索功能——按关键词搜索描述内容**

- **Given**：系统中存在多条反馈，其中 3 条反馈的描述内容包含「CDN」关键词
- **When**：客服在搜索框中输入「CDN」，点击搜索
- **Then**：列表筛选出包含「CDN」的 3 条反馈，描述内容中「CDN」三个字母高亮显示；其他反馈不显示；搜索框保持关键词，点击「清除搜索」恢复完整列表

**验收标准 AC-024：多条件筛选——状态 + 类型组合筛选**

- **Given**：系统中存在多条反馈，包含不同状态和类型
- **When**：客服在筛选栏中选择「状态 = 待处理」「问题类型 = 视频异常」「时间范围 = 近 7 天」三个筛选条件，点击「应用」
- **Then**：列表显示同时满足全部三个条件的反馈，其他反馈不显示；筛选栏中三个筛选条件的值均保持选中状态；列表顶部显示当前筛选条件摘要「待处理 · 视频异常 · 近 7 天」

**验收标准 AC-025：导出功能——按当前筛选条件导出 Excel**

- **Given**：客服应用了「状态 = 已完成」「时间范围 = 近 30 天」的筛选条件，列表显示 127 条符合条件的反馈
- **When**：客服点击右上角「导出」按钮
- **Then**：浏览器弹出文件下载对话框，下载文件名为「反馈列表_已完成_近30天_YYYYMMDD.xlsx」；Excel 文件包含 127 行数据（不含表头），每行包含：反馈编号、用户昵称、问题类型、问题描述（完整）、提交时间、处理状态、操作时间、备注等字段；导出过程后台显示加载提示「正在导出，请稍候...」

### 5.1.4 状态变更通知

**验收标准 AC-026：客服变更状态后发送应用内通知**

- **Given**：用户提交了一条反馈（编号 FB-20260328-001），当前状态为「待处理」，用户已登录客户端
- **When**：客服将该反馈状态变更为「处理中」，点击保存
```

## 第六章、技术约束

### 章节说明

本章描述项目的技术限制、集成依赖、安全合规要求和性能指标，确保开发团队在技术选型和架构设计时充分考虑约束条件，在项目早期识别可能影响进度的技术风险。本章与「需求详述」章节并行填写，产品经理负责提供业务需求侧的要求（性能目标、安全合规要求），技术负责人负责评估可行性并填充具体实现方案。如技术约束与业务需求存在冲突（如性能要求过高导致实现成本不可接受），产品经理和技术负责人应在评审会上协商调整，必要时回溯修改需求详述。

### 填写说明

- **技术栈限制**：明确前端、后端、数据库、缓存、文件存储等技术选型要求，尽量与团队现有技术栈保持一致，降低学习成本和引入新技术的风险。如果有多个可选技术方案，应在备注中说明选择理由和排除理由。
- **集成与依赖**：列出与外部系统或内部其他团队的接口依赖，包括依赖系统的名称、接口类型（REST API / 消息队列 / SDK 等）、对接负责人和预计交付时间。这些依赖项是项目排期的关键输入，任何依赖延迟都会直接影响本项目的里程碑。
- **安全与合规要求**：描述数据安全、隐私合规、审计日志等方面的强制要求。这些要求通常不可协商（尤其是涉及法律法规的合规要求），必须在项目初期就让技术负责人确认实现方案。

- **兼容性要求**：描述浏览器兼容性、设备兼容性、国际化要求、无障碍要求等。这些要求直接影响前端开发的工作量和测试范围，应尽早确认。

```markdown
| 内部备注 | 仅对客服和管理员可见的备注信息，存储在反馈记录的单独字段中，用户端完全不可见，用于客服内部协作和交接 | 对立于「公开回复」，后者是用户可见的处理结果说明 |
| 反馈编号（Feedback ID） | 系统为每条用户反馈生成的唯一标识符，格式为 FB-YYYYMMDD-XXX，用户用于查询反馈处理进度，格式中 XXX 为当日该用户提交的第几条反馈（从 001 起，不足 3 位前补零） | 例：FB-20260328-001 表示 2026 年 3 月 28 日该用户当天提交的第 1 条反馈 |
| P0 / P1 / P2 / Won't | MoSCoW 优先级分类：P0（Must，必须有）是核心功能，缺一不可；P1（Should，应该有）是重要功能，优先实现；P2（Could，可以有）是锦上添花的功能；Won't（这次不做）是明确排除的需求 | 与互联网行业常见的「P0/P1/P2」优先级基本对应，但 Won’t 是 MoSCoW 独有的「明确记录在案但本期不做」类别 |
| Given / When / Then | 行为驱动开发（BDD）中描述验收标准的格式：Given（前提条件）、When（触发操作）、Then（预期结果），迫使产品经理将验收标准写得一目了然 | 源自 Cucumber / Gherkin 语法，最初用于自动化测试用例，后被广泛应用于产品需求验收标准编写 |
| MTTR（Mean Time To Recovery） | 从系统故障发生到服务恢复正常的中位时间，是衡量系统可靠性的重要指标 | SRE 领域核心指标之一，与 SLA 可用性（99.9%）直接关联 |
| P95 / P99 | 百分位响应时间指标：P95 表示 95% 的请求响应时间低于此值，P99 表示 99% 的请求响应时间低于此值 | 与平均值（Mean）相比，百分位指标能更好地反映用户体验的真实上界（平均值容易被极端慢请求拉高）|
| SLA（Service Level Agreement） | 服务等级协议，定义服务提供方承诺的服务质量水平，通常以可用性百分比表示（如 99.9% = 月均停机 < 44 分钟）| SLA 与内部 SLO（Service Level Objective，服务等级目标）不同，SLA 是对外承诺，SLO 是内部追求 |
| WCAG 2.1 AA | 万维网联盟（W3C）发布的《Web 内容无障碍指南》2.1 版 AA 级别，要求内容对残障人士可访问，包括文字替代描述、键盘可操作、足够的颜色对比度等 | 全球多数政府网站的无障碍合规标准 |
| OSS（Object Storage Service） | 对象存储服务，用于存储非结构化数据（如图片、视频、文件），以 RESTful API 访问，提供高可靠性和低成本的大规模存储 | 阿里云 OSS、AWS S3、Google Cloud Storage 等均为此类服务 |
```
